# Docker & Docker Compose Containerization Guide

> Note: the current runnable Docker setup in this repo uses one shared SQL Server container plus RabbitMQ. Some autogenerated sections below still mention the earlier PostgreSQL draft and should be treated as historical notes, not the source of truth. Use `docker-compose.yml` and `docker-compose.prod.yml` as the authoritative configuration.

## Overview

This guide explains how to build and run all microservices and the frontend using Docker and Docker Compose. The containerized setup includes:

- **6 Backend Microservices** (.NET 10.0)
  - Identity Service (Authentication & Authorization)
  - Answer Service
  - Assessment Service
  - Candidate Service
  - Result Service
  - API Gateway (YARP Reverse Proxy)

- **Frontend** (Angular 17 with Nginx)

- **Infrastructure**
  - 1 shared SQL Server Database container
  - RabbitMQ Message Broker (replacing in-memory MassTransit)

## Prerequisites

- **Docker Desktop** (version 20.10+) - [Download](https://www.docker.com/products/docker-desktop)
- **Docker Compose** (included with Docker Desktop)
- **PowerShell 7+** (for running helper scripts)

## Quick Start

### 1. Build All Docker Images

```powershell
./docker-build-all.ps1
```

This builds Docker images for all 6 backend services and the Angular frontend. Images are built in layers and cached for subsequent builds.

**Alternative** - Build only specific services:
```powershell
./docker-build-all.ps1 -Services api-gateway,identity-service -NoBuildCache
```

### 2. Start All Containers

```powershell
./docker-up.ps1
```

This starts all containers in the background and waits for services to become healthy. The script will:
- Start 1 SQL Server container
- Start RabbitMQ
- Start 6 microservices (with automatic database migrations)
- Start the frontend on Nginx
- Display available endpoints

**Output example:**
```
✅ All containers started successfully!

📋 Services available at:
   • Frontend: http://localhost:4200
   • API Gateway: http://localhost:7080
   • API Gateway Health: http://localhost:7080/health
   • RabbitMQ Management: http://localhost:15672 (guest/guest)
   • Identity Service: http://localhost:7140/health
   ...
```

### 3. Verify Services Are Running

```powershell
./docker-status.ps1
```

This displays the status of all containers and their health checks.

### 4. View Logs

```powershell
./docker-logs.ps1                    # All logs
./docker-logs.ps1 -Service api-gateway -Follow
./docker-logs.ps1 -Service identity-service -Tail 50
```

### 5. Stop All Containers

```powershell
./docker-down.ps1
```

To also remove persistent data (databases):
```powershell
./docker-down.ps1 -RemoveVolumes
```

### Docker Network

Services communicate via Docker's internal bridge network (`app-network`). Services use container names as hostnames (e.g., `identity-service`, `api-gateway`).

## Configuration

### Environment Variables (`.env` file)

The `.env` file at the project root contains all configuration:

```env
# SQL Server credentials (change in production!)
SQL_SA_PASSWORD=SqlPassword123!

# RabbitMQ credentials
RABBITMQ_USER=guest
RABBITMQ_PASSWORD=guest

# Environment mode
ASPNETCORE_ENVIRONMENT=Development
```

**Important**: Change these credentials in production!

### Configuration Files

Each service has configuration in its `appsettings.json`:

- **appsettings.json** - Local development defaults
- **appsettings.Development.json** - Dev-specific overrides with environment variable placeholders
- **Program.cs** - Reads configuration from `appsettings` and environment variables

**Example (answer-service/Api/appsettings.Development.json):**
```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Host=${DB_HOST:localhost};Port=${DB_PORT:5432};Database=${DB_NAME:answer_db}..."
  },
  "RabbitMQ": {
    "Host": "${RABBITMQ_HOST:localhost}",
    "Port": "${RABBITMQ_PORT:5672}",
    "Username": "${RABBITMQ_USER:guest}",
    "Password": "${RABBITMQ_PASSWORD:guest}"
  }
}
```

## Database Initialization

### Automatic Migration

Each service automatically runs Entity Framework migrations on startup:

```csharp
using (var scope = app.Services.CreateScope())
{
    var dbContext = scope.ServiceProvider.GetRequiredService<YourDbContext>();
    dbContext.Database.Migrate();  // Runs pending migrations
}
```

Migrations are executed in this order during container startup:
1. SQL Server container starts and becomes healthy
2. Microservice container starts
3. EF Core runs any pending migrations automatically

### Manual Database Operations

Connect to a service's database:

```powershell
# Connect to SQL Server
docker exec -it sqlserver /opt/mssql-tools18/bin/sqlcmd -S localhost -U sa -P SqlPassword123!

# List databases
SELECT name FROM sys.databases;

# Select a database
USE AnswerServiceDb;

# List tables
SELECT TABLE_NAME FROM INFORMATION_SCHEMA.TABLES;

# Run a query
SELECT TOP 10 * FROM YourTable;

# Exit
QUIT
```

## Messaging & Events

### RabbitMQ Setup

Services use **MassTransit 8.0** with RabbitMQ instead of in-memory messaging:

```csharp
builder.Services.AddMassTransit(x =>
{
    x.AddConsumer<YourEventConsumer>();
    
    x.UsingRabbitMq((context, cfg) =>
    {
        cfg.Host(rabbitmqHost, rabbitmqPort, "/", h =>
        {
            h.Username(rabbitmqUser);
            h.Password(rabbitmqPassword);
        });
        cfg.ConfigureEndpoints(context);
    });
});
```

### RabbitMQ Management UI

Access RabbitMQ management at:
- **URL**: http://localhost:15672
- **Username**: guest
- **Password**: guest

Here you can:
- Monitor message queues
- View exchanges and bindings
- Create/delete queues
- Monitor consumer performance

## Frontend Setup

### Build Process

The Angular frontend uses a **multi-stage Docker build**:

1. **Build Stage**: Node.js container builds Angular app
   - Installs dependencies from `package.json`
   - Runs `npm run build` (creates optimized distribution)

2. **Runtime Stage**: Nginx serves built static files
   - Copies built artifacts from build stage
   - Serves on port 80 with proper routing
   - Health check endpoint at `/health`

### API Proxy Configuration

The frontend's proxy configuration (`proxy.conf.json`) routes API calls to the API Gateway:

```json
{
  "/api": {
    "target": "http://api-gateway:80",  // Uses Docker container hostname
    "secure": false,
    "changeOrigin": true
  },
  "/hubs": {
    "target": "http://api-gateway:80",  // SignalR WebSocket connections
    "ws": true
  }
}
```

**During local development** (without Docker), the proxy still works:
```json
{
  "/api": {
    "target": "http://localhost:7080",  // Local API Gateway
    "secure": false,
    "changeOrigin": true
  }
}
```

## Troubleshooting

### Services Won't Start

**Check logs:**
```powershell
./docker-logs.ps1 -Service api-gateway -Follow
```

**Common issues:**
- Port already in use: `lsof -i :7080`
- Insufficient resources: Check Docker Desktop resources
- Database migration failed: Check database connectivity

### Can't Connect to Services

1. **Verify containers are running:**
   ```powershell
   ./docker-status.ps1
   ```

2. **Check network connectivity:**
   ```powershell
   docker network inspect app-network
   ```

3. **Test service health:**
   ```powershell
   curl http://localhost:7080/health
   curl http://localhost:7140/health
   ```

### Database Connection Issues

**Check database is running:**
```powershell
docker exec sqlserver /opt/mssql-tools18/bin/sqlcmd -S localhost -U sa -P SqlPassword123! -Q "SELECT 1"
```

**Check connection string:**
- Verify credentials match in `.env` and container environment
- Verify hostname is correct (e.g., `sqlserver`, not `localhost`)

### RabbitMQ Issues

**Check RabbitMQ is running:**
```powershell
docker logs rabbitmq
```

**Verify connectivity:**
```powershell
docker exec answer-service curl -i -u guest:guest http://rabbitmq:15672/api/vhosts
```

### Frontend Not Loading

1. **Check Nginx logs:**
   ```powershell
   ./docker-logs.ps1 -Service frontend -Follow
   ```

2. **Verify API Gateway is accessible:**
   ```powershell
   curl http://localhost:7080
   ```

3. **Check CORS configuration in services** - may need to adjust allowed origins

## Development Workflow

### Local Development (Without Docker)

For faster development iteration:

```powershell
# Terminal 1: Run backend services locally
cd backend
./start-all-services.ps1

# Terminal 2: Run frontend
cd frontend
npm start
```

This is faster for development as you avoid container build times.

### Hot Reload in Containers

Containers don't support hot reload by default. For development:

1. Use local development mode (above) or
2. Rebuild and restart specific services:

```powershell
./docker-build-all.ps1 -Services api-gateway
./docker-up.ps1
```

## Production Considerations

### Security

For production, update:

1. **Database credentials** (`.env` and deployment)
2. **RabbitMQ credentials** (use strong passwords)
3. **JWT signing key** in `IdentityService.Api.appsettings.json`
4. **CORS allowed origins** (specify actual domain, not `*`)

Example production `.env`:
```env
ANSWER_DB_PASSWORD=<strong-random-password>
ASSESSMENT_DB_PASSWORD=<strong-random-password>
...
RABBITMQ_USER=<custom-user>
RABBITMQ_PASSWORD=<strong-random-password>
JWT_KEY=<long-random-key>
ASPNETCORE_ENVIRONMENT=Production
```

### SSL/TLS Certificates

Current setup uses HTTP. For production:

1. Add Nginx reverse proxy with SSL termination:
   ```yaml
   nginx-ssl:
     image: nginx:latest
     volumes:
       - ./certs:/etc/nginx/certs:ro
     ports:
       - "443:443"
   ```

2. Or use cloud provider's load balancer (AWS ALB, Azure AppGateway, etc.)

### Docker Image Registry

For cloud deployment, push images to a registry:

```powershell
# Build and tag images
docker build -t myregistry.azurecr.io/answer-service:latest backend/answer-service/Api/

# Push to registry
docker push myregistry.azurecr.io/answer-service:latest

# Update docker-compose.yml to use registry images
```

## Performance Optimization

### Multi-Stage Builds

All Dockerfiles use multi-stage builds to reduce final image size:
- First stage: Compile .NET code
- Second stage: Runtime with only compiled binaries (no source)

### Caching Strategies

Docker caches build layers. Order matters:

```dockerfile
# Good - package files change rarely
COPY *.csproj .
RUN dotnet restore

# Then copy source (changes frequently)
COPY . .
RUN dotnet build
```

### Resource Limits

Add resource limits in production `docker-compose.prod.yml`:

```yaml
services:
  api-gateway:
    deploy:
      resources:
        limits:
          cpus: '1'
          memory: 512M
        reservations:
          cpus: '0.5'
          memory: 256M
```
